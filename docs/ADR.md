# Architecture Decision Records

## 철학
MVP 속도와 운영비 0원을 최우선. 서버를 두지 않고, 가능한 한 공식·합법 경로를 쓰며,
"작동하는 최소 구현"을 선택한다. 비공식/취약한 요소(메신저봇R)는 격리하고 문서에 리스크를 명시한다.

---

### ADR-001: 봇 런타임으로 메신저봇R 선택
**결정**: 카카오톡 오픈채팅 자동화에 메신저봇R(안드로이드, Rhino JS)을 사용.
**이유**: 카카오 공식 채널/오픈빌더는 오픈채팅방 자동 푸시를 지원하지 않는다. 오픈채팅방에
자동으로 메시지를 보낼 수 있는 현실적 수단은 메신저봇R이 사실상 유일하다.
**트레이드오프**: 비공식(약관 리스크) · 카카오 업데이트 시 깨질 수 있음 · 폰 24h 구동 필요 ·
Rhino라 ES5 위주 문법 제약.

### ADR-002: 수집기를 GitHub Actions cron으로 (서버리스)
**결정**: 채용공고 수집을 GitHub Actions 스케줄 워크플로우로 돌리고, 결과를 repo에 커밋한다
(Simon Willison "git scraping" 패턴).
**이유**: 24시간 서버를 띄우지 않아도 무료로 주기 수집이 가능. 결과 JSON을
raw.githubusercontent.com으로 그대로 정적 서빙 → 별도 API 서버도 불필요.
**트레이드오프**: cron 최소 주기·지연(분 단위) 한계 · public repo 필요 · 실시간성은 낮음(채용공고엔 충분).

### ADR-003: 데이터 소스로 사람인 공식 API 선택 (크롤러 재사용 대신)
**결정**: 사람인 공식 JSON API(`oapi.saramin.co.kr/job-search`)를 requests로 호출.
**이유**: IT 채용 크롤러(원티드/점핏: `heehehe/job-trend`, `Joylish/wanted-crawler` 등)는
전부 Selenium 기반이라 무료 서버리스(Actions)에서 무겁고 취약하다. 원티드는 공개 API도 없다.
공식 API는 requests 한 번이면 되고 합법·안정적이라 Actions에 완벽히 맞는다.
**트레이드오프**: 사람인만 커버(원티드/점핏 제외) · access-key 발급 필요 · 1일 500회 호출 제한 ·
IT 커버리지가 부족하면 워크넷 공식 API 추가로 확장 가능.

### ADR-004: 정적 jobs.json을 "API"로 사용
**결정**: 수집 결과를 `data/jobs.json`으로 커밋하고, 봇은 raw URL로 이 파일을 읽는다.
**이유**: DB·API 서버 없이 무료 정적 호스팅으로 데이터 제공. git 히스토리로 변경 추적도 됨.
**트레이드오프**: 쓰기 동시성/대용량엔 부적합(채용공고 규모엔 무관) · 갱신은 커밋 주기에 묶임.
